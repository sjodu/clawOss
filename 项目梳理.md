# ClawOSS 项目梳理文档

## 1. 项目概述

**项目名称**: ClawOSS  
**版本**: 0.1.0 (v6)  
**类型**: 自主开源贡献 AI Agent  
**核心引擎**: OpenClaw  
**GitHub 账号**: @BillionClaw  
**仓库地址**: https://github.com/billion-token-one-task/ClawOSS

### 1.1 项目定位

ClawOSS 是一个基于 OpenClaw 平台构建的自主 AI Agent，能够 24/7 无人值守地：
- 自动发现 GitHub 开源项目中的问题（issues）
- 实现修复方案（bug fixes, docs fixes, typo fixes, test additions）
- 提交 Pull Request
- 处理代码审查反馈
- 追踪 PR 合并状态

**核心理念**: "OpenClaw 是引擎，ClawOSS 是赛车"

### 2.2 核心组件

#### 2.2.1 编排器 (Orchestrator)
- **会话**: main session
- **职责**: 
  - 健康检查与上下文管理
  - 工作队列管理
  - 子代理生命周期管理
  - 结果收集与处理
  - 自我唤醒循环

#### 2.2.2 常驻子代理 (4 个槽位)
1. **Scout** (标签: scout-*)
   - 持续发现 GitHub issues
   - 输出: `memory/work-queue-staging.md`
   
2. **PR Monitor Scan** (标签: pr-monitor-scan)
   - 快速扫描开放 PR
   - 处理简单操作（合并、关闭、提醒）
   - 输出: `memory/pr-monitor-active.md`
   
3. **PR Monitor Deep** (标签: pr-monitor-deep)
   - 深度分析 PR 评论
   - 构建跟进上下文
   - 输出: `memory/followup-staging.md`
   
4. **PR Analyst** (标签: pr-analyst)
   - 投资组合分析
   - 失败模式识别
   - 策略建议
   - 输出: `memory/pr-strategy.md`, `memory/repo-blocklist.md`

#### 2.2.3 实现子代理 (10 个槽位)
- **新 PR 优先**: 先填满所有 10 个槽位用于新实现
- **跟进次之**: 仅当所有槽位满或无新工作时处理跟进
- **工作流程**: 克隆 → 理解 → 复现 → 修复 → 测试 → 审查 → 提交 → 清理
- **结果输出**: `memory/subagent-result-*.md` (YAML frontmatter)

## 3. 自主循环 - HEARTBEAT.md

### 3.1 循环触发
- **频率**: 每 5 分钟
- **触发方式**: OpenClaw Gateway 心跳
- **模式**: 无限循环，永不空闲
- **原则**: 总有工作可做 - 如果队列空，运行发现；如果无新问题，处理跟进；如果无跟进，扩展到新领域

### 3.2 七步循环

#### 步骤 0: 健康检查
- **0a. 上下文检查**: 使用 `session_status` 工具
  - 如果上下文 > 35%: 立即压缩
  - 压缩前刷新状态到内存文件
- **0b. 熔断器**: 读取 `wake-state.md`
  - 如果 errors_this_hour >= 5: 暂停 2 分钟后继续
- **0c. Dashboard 自检**: 
  - 获取指令、避免仓库列表、开放 PR 仓库列表
  - 超时则跳过，不阻塞循环

#### 步骤 0.5: 常驻子代理管理
- 检查 4 个常驻子代理状态（通过 `sessions_list`）
- 如果死亡或缺失: **立即重生**
- 重生流程:
  1. 从磁盘读取模板文件（`templates/subagent-*.md`）
  2. 使用 `sessions_spawn` 启动
  3. 传递必要的附件（trust-repos.md, pr-ledger.md 等）

#### 步骤 1: 停滞恢复
- 检测停滞的子代理（> 5 分钟无消息）
- 杀死停滞代理
- 将任务重新加入队列顶部
- 2 次连续停滞 = 跳过该任务

#### 步骤 2: 选择新工作
- **2a. 合并暂存**: 合并 `work-queue-staging.md` 和 `followup-staging.md` 到 `work-queue.md`
- **2b. 按 P(merge) 排序**: 合并概率 >= 60 的优先
- **2c. 计数与选择**:
  - 如果活跃子代理 >= 10: 跳到步骤 6
  - 如果活跃子代理 < 10: 选择下一个任务
  - 如果队列 < 5: 立即运行 `oss-discover`
- **2d. 门控检查**:
  - 去重检查（pr-ledger.md, 锁文件）
  - 黑名单硬阻止
  - 类型检查（仅 bug/docs/typo/test）
  - 标题拒绝（feature/enhance/refactor 等关键词）
  - 健康门控（stars >= 100, 非归档, 30 天内有推送）
  - 超越检查（已分配、已有 PR、已声明）
  - 已修复检查（避免重复提交）

#### 步骤 3: 分类（< 3 分钟）
- 快速健康检查
- 类型验证
- 运行 `oss-triage` 技能
- 超越检查
- 评分（1-25 分）
- 使用 `web_search` 进行快速研究

#### 步骤 4: 生成实现子代理
- **4a. 预生成评论**（分数 >= 8）: 在 issue 上发布简短评论
- **4b. 生成子代理**:
  1. 从磁盘读取 `templates/subagent-implementation.md`
  2. 替换占位符（repo, issue, title）
  3. 使用 `sessions_spawn` 启动
  4. 立即标记为 `spawned_pending`
- **4c. 传递上下文**: 开放 PR 列表、仓库约定、issue 详情

#### 步骤 5: PR 跟进（仅当所有 10 个实现槽位已满）
- 读取 `memory/followup-staging.md`
- 加载 `templates/subagent-followup.md`
- 使用暂存数据作为附件生成

#### 步骤 6: 处理子代理结果
- **6a. 实现结果**: 列出 `memory/subagent-result-*.md`
  - 成功 + PR URL: 标记为 `completed`，更新账本
  - 成功无 PR URL: 标记为 `failed`，重新排队一次
  - 失败/放弃: 记录失败原因
- **6b. 跟进结果**: 列出 `memory/subagent-result-followup-*.md`
  - 根据状态更新轮次
  - 第 3 轮: 礼貌脱离
- **6c. 信任验证**: 更新 `trust-repos.md` 时验证合并计数

#### 步骤 6.5: 强制上下文检查
- 使用 `session_status` 检查上下文
- 如果 > 35%: 刷新状态并压缩
- **不要跳过此步骤** - 防止下一个循环超时

#### 步骤 7: 报告、清理与循环
- 运行 `dashboard-reporter`
- 更新 `wake-state.md`
- 删除已处理的结果文件
- 清理过期的工作队列项（> 30 天）
- **自我唤醒**: `openclaw system event --text "cycle-complete" --mode now`
- **立即循环回步骤 2** - 永不等待

### 3.3 关键原则
- **永不空闲**: 禁止说"监控完成事件"或"待命"
- **新 PR 优先**: 先填满 10 个实现槽位，然后才处理跟进
- **懒加载**: 仅读取当前步骤需要的内存文件
- **反死锁**: 允许每个仓库多个开放 PR
- **永不等待**: 每步之后继续下一步

## 6. PII 清理器插件 - 内容过滤绕过

### 6.1 问题背景
OpenRouter 的内容过滤器会拦截包含 `@` 符号的请求，导致 403 错误。这对 GitHub 操作（如 `@BillionClaw`）造成严重影响。

### 6.2 解决方案
`plugins/pii-sanitizer/index.js` (101 行) 实现双向 `@` 符号替换：

```javascript
var FULLWIDTH_AT = '＠'   // ＠ (全角 @)

// 流程:
// 磁盘 (@) ──读取──▶ 会话 (＠) ──模型──▶ 工具调用 (＠→@) ──▶ 磁盘 (@)
```

### 6.3 钩子注册
```javascript
api.on('tool_result_persist')   → sanitize(@ → ＠)   // 文件读取
api.on('before_message_write')  → sanitize(@ → ＠)   // 助手输出
api.on('before_tool_call')      → desanitize(＠ → @) // 写入/编辑/执行
```

### 6.4 效果
- OpenRouter 永远看不到原始 `@` 符号
- **零 403 内容过滤错误**
- 对代理透明，无需修改提示词

### 7.2 功能模块

#### 7.2.1 实时数据流
- **心跳**: 每个循环的状态更新
- **指标**: Token 使用量 + 成本追踪
  - 输入: $0.60/M tokens
  - 输出: $3.00/M tokens
- **对话**: 实时对话流（编排器 + 每个子代理标签页）
- **状态**: 工作队列 + 流水线状态 + 活跃仓库
- **日志**: 结构化审计日志

#### 7.2.2 API 端点
```
/api/ingest/heartbeat     ← 状态、运行时间、仓库

/api/ingest/conversation  ← sessionId、role、content、toolName
/api/ingest/state         ← workQueue、pipelineState、activeRepos
/api/ingest/logs          ← 结构化审计日志
/api/agent/health-check   → directives、avoidRepos、reposWithOpenPRs
```

### 7.3 遥测管道

#### 7.3.1 dashboard-reporter (628 行)
- **状态累积**: inputTokens, outputTokens, toolCallCount, reposUsed
- **事件处理**:
  - `user_message`: 发布对话
  - `after_tool_call`: 累积 tokens + 检测 sessions_spawn
  - `agent_end`: 刷新指标 + 心跳 + 状态 + 日志
- **网络**: 非阻塞 POST，2 次尝试，10 秒超时

#### 7.3.2 audit-logger (133 行)
- 即发即弃的结构化日志
- 事件: command:new, after_tool_call, agent_end
- 10 秒超时

#### 7.3.3 dashboard-sync.sh (319 行)
- 后台进程，每 10 秒轮询 JSONL
- 构建会话映射（Python3 读取 sessions.json）
- 解析内容块（text/thinking/toolCall/tool_result）
- POST 到 `/api/ingest/conversation`
- 每 60 秒自我更新（md5 检查）

#### 7.3.4 pr-ledger-sync.sh (185 行)
- 每 60 秒运行（launchd）
- **数据源 1**: `gh search prs --author BillionClaw --limit 200`
- **数据源 2**: `grep subagent-result-*.md` 提取 PR URL
- Python 合并器：GitHub 是状态的权威来源
- 输出: `workspace/memory/pr-ledger.md` (排序的 markdown 表格)

```bash

步骤 4  → ln -sf workspace → ~/.openclaw/workspace
```

## 10. 贡献策略

### 10.1 核心理念
**优化合并率，而非提交数量**

- 50 个未审查的 PR = 0 影响
- 1 个合并的 typo 修复 > 1 个未审查的 bug 修复
- **深度优于广度**: 在 1 个仓库 3+ 个合并的 PR > 在 30 个仓库 30 个未审查的 PR

### 10.2 贡献类型（按合并概率排序）
1. **Typo 修复** - 几乎保证合并
2. **文档修复** - 高合并率
3. **测试添加** - 良好合并率
4. **Bug 修复** (good-first-issue/help-wanted) - 维护者需要帮助
5. **Bug 修复** (labeled bug/defect/regression) - 已确认的 bug

### 10.3 不在范围内
- ❌ 功能添加
- ❌ 重构
- ❌ 依赖更新
- ❌ 性能优化
- ❌ 增强
- ❌ > 30 天的 issues
- ❌ < 200 stars 的仓库
- ❌ 未通过健康门控的仓库

### 10.4 贡献组合
**混合策略**: 60% 简单胜利 + 40% 实质性 bug 修复

### 10.5 信任建设策略
- **返回赢家**: 如果仓库合并了我们的 PR，它是下一次贡献的首选目标
- **追踪关系**: 维护者积极参与（批准、感谢、反馈）的仓库排在队列顶部
- **快速放弃失败者**: 如果仓库在 24 小时内关闭我们的 PR 而不审查，降低优先级 30 天
- **专注 10-15 个仓库**: 建立声誉作为可信贡献者

### 10.6 发现策略

#### 10.6.1 发现领域（轮换所有 - AI 领域已饱和）
1. **Agentic AI**: `topic:llm`, `topic:agent`, `topic:rag`, `topic:ai` + `stars:>200`
2. **开发者工具**: `topic:cli`, `topic:devtools`, `topic:developer-tools`, `topic:terminal`
3. **Web 框架**: `topic:web-framework`, `topic:nextjs`, `topic:fastapi`, `topic:django`
4. **数据库与存储**: `topic:database`, `topic:sql`, `topic:nosql`, `topic:vector-database`
5. **云原生**: `topic:kubernetes`, `topic:docker`, `topic:cloud-native`
6. **测试与质量**: `topic:testing`, `topic:linting`, `topic:code-quality`
7. **数据工程**: `topic:data-pipeline`, `topic:etl`, `topic:data-engineering`

#### 10.6.2 新鲜度层级
1. **热门** (< 3 天): 最高优先级
2. **最近** (3-14 天): 良好候选
3. **老化** (14-30 天): 仅限简单任务
4. **过期** (> 30 天): 跳过

### 10.7 评分系统

#### 10.7.1 质量分数 (1-25)
- +5: docs/typo 修复
- +3: 测试添加
- +5: 平均合并时间 < 3 天
- +3: 审查率 > 80%
- +2: good-first-issue/help-wanted 标签
- -5: 平均合并时间 > 14 天
- -10: 我们的 PR 100% 关闭率
- **跳过**: 30 天内 0 次合并

#### 10.7.2 P(merge) - 合并概率 (0-100)
加权公式考虑：
- **信任** (25%): 仓库是否在 trust-repos.md 中
- **大小** (20%): LOC 变更（25-100 最佳）
- **任务类型** (15%): typo > docs > test > bug
- **仓库响应性** (15%): 合并速度、审查率
- **新鲜度** (10%): issue 年龄
- **贡献者契合度** (10%): 我们的历史成功率
- **竞争** (5%): 其他开放 PR

**阈值**:
- P(merge) >= 30: 尝试
- P(merge) >= 60: 优先生成

## 11. 实现工作流

### 11.1 Bug 修复（复现优先）
1. **确认** - 验证是有效贡献（bug/docs/typo/test）
2. **理解** - 阅读架构，追踪执行路径，理解根本原因
3. **复现** - 编写失败测试演示 bug
4. **实现** - 全面修复解决根本原因
5. **验证** - 失败测试现在通过，无回归
6. **审查** - 自检：完全解决？根本原因还是症状？
7. **提交** - 带有根本原因分析和复现证据的 PR

### 11.2 文档/Typo 修复
识别 → 对照代码验证 → 最小化修复 → 自检准确性 → 提交

### 11.3 测试添加
识别未测试路径 → 编写测试（仓库约定）→ 验证通过 → 提交

### 11.4 放弃规则
- 实现期间发现是功能请求: 立即放弃
- 太复杂无法完全解决: 放弃（不做部分修复）
- 10 分钟内无进展: 放弃
- 2 次尝试后测试失败: 放弃

## 12. PR 跟进生命周期

### 12.1 检测
- PR Monitor Scan (常驻) 快速扫描所有开放 PR
- PR Monitor Deep (常驻) 深度分析评论
- 暂存到 `memory/followup-staging.md`

### 12.2 分类
- **简单操作**: 合并、关闭、提醒（Monitor Scan 自动处理）
- **代码变更**: 需要实现（暂存给主编排器）

### 12.3 实现
子代理工作流：
1. 克隆仓库
2. 检出 PR 分支
3. 读取所有评论（内联 + 顶层 + 正式审查）
4. 实现变更
5. 运行测试
6. 推送到同一分支
7. 通过 `gh pr comment` 和 `gh api` 响应
8. 写入结果
9. 清理

### 12.4 审查者沟通
- 感谢一次（响应顶部），专业简洁
- 永不争论 - 实现请求
- "不合适" / "超出范围": 调整范围并重做
- "修复不起作用" / "错误方法": 用不同方法重做，强制推送到同一分支
- 范围扩展请求: 接受
- 永不 ping 或请求重新审查
- **身份问题**（"你是 AI 吗？"）: "我是 BillionClaw。" 然后重定向到贡献。无需进一步解释。

### 12.5 轮次限制
- 最多 3 轮跟进
- 第 3 轮后: 礼貌脱离，留 PR 给维护者

## 13. 质量标准

### 13.1 通用标准
- 每个 PR 必须完全解决其范围 - 不做部分修复
- 代码风格必须匹配目标仓库约定
- 提交消息: `fix(scope): desc`, `docs(scope): desc`, `test(scope): desc`
- 无 AI 痕迹: 无不必要的注释，无过度工程，无"我"陈述，无通用措辞
- CI 矩阵检查强制: 提交前阅读 `.github/workflows/`

### 13.2 Bug 修复标准
- 根本原因分析 + 复现证据
- 失败测试（修复前）→ 通过测试（修复后）

### 13.3 文档/Typo 标准
- 对照实际代码行为验证正确性

### 13.4 测试标准
- 遵循仓库测试约定
- 覆盖有意义的未测试路径

## 19. 去重与锁定机制

### 19.1 去重检查
在生成前检查：
1. `memory/pr-ledger.md` - 已提交的 PR
2. `memory/impl-spawn-state.md` - spawned_pending 标记
3. `memory/locks/{owner}_{repo}.lock` - 仓库锁文件
4. `gh search prs --author BillionClaw --repo {owner}/{repo}` - 实时 GitHub 检查

### 19.2 锁文件
```bash
# 生成前创建
echo "$(date -u +%Y-%m-%dT%H:%M:%SZ) {issue}" > memory/locks/{owner}_{repo}.lock

# 子代理在 PR 创建或失败后删除
# 编排器在步骤 1 清理过期锁（> 30 分钟）
```

### 19.3 超越检查
在开始任何 issue 之前验证：
1. **无链接 PR**: 检查 issue 时间线的交叉引用 PR
2. **未分配**: 检查 issue 受让人
3. **无竞争 PR**: 搜索其他贡献者在同一 issue 上的开放 PR
4. **无文件冲突**: 读取仓库中的开放 PR，检查文件重叠
5. **未修复**: 检查最近的提交和合并的 PR

## 20. 失败处理

### 20.1 失败类别
所有失败使用 `templates/subagent-result-schema.md` 中的标准 `failure_reason` 类别：
- `repo_health_fail` - 缓存 24 小时
- `ci_incompatible` - CI 损坏（不是我们的）
- `api_rate_limited` - 退避
- `fix_rejected_terminal` - 2+ 次失败的重做
- `reviewer_rejected_scope` - 范围被拒绝
- `already_fixed_upstream` - 已在上游修复
- `superseded` - 已分配或有竞争 PR
- `model_error` - 重试一次然后跳过

### 20.2 失败日志
- 追踪在 `memory/failure-log.md`
- 同一类别/天 3+ 次触发策略调整

### 20.3 仓库降级
- `fix_rejected_terminal` (2+ 次失败的重做): 降级 30 天
- `reviewer_rejected_scope`: 降级 30 天
- 单个 `fix_rejected`: 重做机会，不降级
- 24 小时内关闭无审查: 降级 30 天
